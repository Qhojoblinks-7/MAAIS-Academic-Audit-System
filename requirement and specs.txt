NTRODUCTION AND REQUIREMENTS SPECIFICATION
1.1 Introduction
The Mando SHTS Academic Audit and Intervention System is an integrated web-based platform designed to modernize the management of student academic records, particularly within the Technical and Vocational departments. In the current educational landscape, the transition to the WAEC Student Transcript Portal (STP) requires a high degree of data integrity and specific formatting that manual systems cannot easily provide.
This system aims to bridge the gap between traditional classroom grading and national digital standards by implementing a "Security-First" approach to grade entry, ensuring that every modification is tracked and every student performance drop is flagged for intervention.
1.2 Problem Statement
Mando SHTS currently faces three primary challenges in its academic processing:
1.	Active Manipulation Risk: Without a digital audit trail, grades can be altered after submission without accountability, leading to potential academic fraud.
2.	Complex Weighting: Technical subjects require complex calculations involving Workshop Practicals (WIP), Technical Drawing, and Theory. Manual calculations are prone to human error.
3.	Lack of Proactive Intervention: There is no automated system to alert the Head of Department (HOD) when a student’s performance declines significantly across terms, often resulting in late-stage failures that could have been prevented.
1.3 System Objectives
The primary objectives of this project are:
●	To develop a Mandatory Audit Trail that logs the "Who, When, and Why" of every grade change.
●	To automate the 30/70 and Technical Weighting calculations for Core and Elective subjects.
●	To provide an Intervention Engine that flags students with a 15% or greater performance drop.
●	To streamline the WAEC STP Export process through an HOD-controlled "Final Lock" mechanism.
1.4 Functional Requirements (FR)
Functional requirements define the specific behaviors and services the system must provide.
●	FR1: Role-Based Access Control (RBAC): The system shall provide distinct interfaces for System Admins, HODs, Teachers, and Students.
●	FR2: Hybrid Grade Entry: The system shall support diverse grading models, including milestone-based marks for workshop projects (Marking out, Assembly, Finishing).
●	FR3: Automated Audit Middleware: Any edit to a saved grade shall trigger a mandatory justification popup; the system shall block the save until a reason is provided.
●	FR4: Behavioral & Safety Logging: Teachers shall be able to log workshop safety compliance and attitudinal ratings (1-5).
●	FR5: Finalization & Locking: The system shall allow HODs to "Lock" results, making them read-only and enabling the WAEC CSV export.
1.5 Non-Functional Requirements (NFR)
Non-functional requirements define the system's operational constraints and quality attributes.
●	NFR1: Data Integrity: The system must utilize ACID-compliant transactions (PostgreSQL) to ensure that result updates and audit logs are saved simultaneously or not at all.
●	NFR2: Security: All passwords must be hashed using industry-standard algorithms (Bcrypt), and all data transmission must be encrypted via SSL/TLS.
●	NFR3: Performance: The system shall generate a departmental broadsheet or WAEC export file in under 5 seconds.
●	NFR4: Usability: The interface must be responsive, allowing teachers to enter marks using tablets or smartphones within the workshop environment.
1.6 User Roles and Responsibilities
Role	Responsibility
System Admin	Manages user accounts, academic years, and system backups.
HOD	Audits grade changes, reviews intervention alerts, and locks finalized data for WAEC.
Teacher	Inputs raw scores, provides justifications for edits, and logs student behavior.
Student	Views personal academic trends and receives counseling notifications.
SYSTEM ANALYSIS AND MODELING
2.1 Overview of System Modeling
To ensure the Mando SHSTS Audit System is both secure and efficient, this chapter employs Data Flow Diagrams (DFDs) and Sequence Diagrams. These models illustrate the journey of a grade from initial entry by a teacher, through the audit middleware, to final approval by the Head of Department.
2.2 Context Diagram (Level 0 DFD)
The Context Diagram represents the entire system as a single process and identifies its relationships with external entities.
 
●	Entities: Teacher, HOD, Student, System Admin, and the WAEC STP Portal.
●	Key Flows: Teachers provide marks; HODs provide approval commands; the System provides processed reports to students and formatted CSV files to the WAEC Portal.
2.3 Data Flow Diagram (DFD) Level 1
The Level 1 DFD breaks down the system into its core functional processes.
 
1.	Process 1.0: User Authentication: Validates credentials and assigns the correct dashboard (Admin/HOD/Teacher/Student).
2.	Process 2.0: Result Management: Handles the logic of 30/70 core grading and multi-milestone technical grading.
3.	Process 3.0: Audit & Validation Engine: The "Watchdog" process that intercepts edits, requests justifications, and writes to the audit_logs table.
4.	Process 4.0: Intervention Logic: Analyzes data trends and generates flags for students with declining performance.
5.	Process 5.0: Report & Export Generation: Consolidates data for student viewing and formats the final HOD-locked data for WAEC.
2.4 Sequence Diagram: "Edit Grade with Audit Trail"
The Sequence Diagram focuses on the timing and interaction between the system's components during the most critical operation: Modifying an existing grade.
 
●	The Handshake:
1.	Teacher clicks edit on the Frontend (React).
2.	Frontend requests current status from Backend (Laravel).
3.	Backend checks Database for a "Locked" status.
4.	If unlocked, the Teacher submits the new mark and a Reason.
5.	Backend performs an atomic transaction: Updating the Results table while simultaneously creating an entry in the Audit_Logs.
6.	Database confirms success back to the user.
2.5 Logic Specification (Decision Table)
To ensure the "Active Manipulation" of grades is stopped, the system follows this strict logic for every data entry:
Condition	Action
If Grade is NEW	Allow Save; Set Status to 'Draft'.
If Grade is CHANGED	Trigger Audit Popup; Require Justification; Log Old/New values.
If Status is 'LOCKED'	Block all Edit/Delete requests; Return "Read-Only" error.
If Change > 15% Drop	Trigger record in intervention_alerts table.
SYSTEM DESIGN AND ARCHITECTURE
3.1 3-Tier Architecture Design
The Mando SHSTS Audit System is built on a 3-Tier Architecture, which separates the user interface, the business logic, and the data storage. This ensures that if the interface changes, the core grading logic remains secure.
1.	Presentation Tier (Frontend): Developed using React.js and Tailwind CSS. This layer handles user interaction and displays the 12 screens for Admin, HOD, Teacher, and Student roles.
2.	Application Tier (Backend): Built with the Laravel (PHP) Framework. This tier contains the "Business Rules," such as the 30/70 weighting logic and the Audit Middleware that intercepts grade changes.
3.	Data Tier (Database): A PostgreSQL relational database that stores all student records, teacher logs, and encrypted audit trails.
 
3.2 Entity-Relationship Diagram (ERD)
The ERD defines the structure of our database. To ensure complete auditability and WAEC compliance, the system utilizes 12 interconnected tables.
●	User Cluster: Users, Teachers, Students, Departments.
●	Academic Cluster: Subjects, Classes, Enrollments.
●	Grading Cluster: Core_Results, Technical_Results.
●	Audit Cluster: Audit_Logs, Intervention_Alerts, System_Settings.

3.4 Security Design
To prevent "Active Manipulation," the design includes:
1.	Row-Level Security: Teachers can only see students in their assigned class_sections.
2.	Immutable Audit Logs: The Audit_Logs table is "Insert-Only." No user (not even the Admin) can delete an audit record once it is created.
3.	Token-Based Authentication: Using Laravel Sanctum, every request from the frontend must carry a secure, timed token.

4.3 Core Logic Implementation
4.3.1 The Audit Middleware (The "Logic Gate")
This is the most critical code block. When a PATCH request is sent to the results table, the system executes the following:
1.	Verify the user is authenticated.
2.	Check if the record is_locked.
3.	If a previous value exists, require a justification string.
4.	Write the old_value, new_value, and reason to the audit_logs table.
________________________________________
4.3.2 The Intervention Engine
The system runs a backend job that compares a student's current total with their average from the previous two terms. If Current_Score < (Previous_Avg * 0.85), the system automatically creates a record in the intervention_alerts table for the HOD to see.
